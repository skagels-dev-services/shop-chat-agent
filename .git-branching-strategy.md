# Branching Strategy for Agenscience Development 

**Recommendation from Claude Code, May 8, 2026:**

You want a workflow that lets you keep building features locally on top of Shopify’s reference repo, while cleanly absorbing upstream updates and avoiding any accidental pushes there. I’ll lay out a practical Git setup with concrete commands and a low-maintenance branching strategy you can use immediately.

Best approach: treat Shopify’s repo as an upstream source, and keep all your enhancements in local-only branches with a protected no-push setup.

Recommended workflow

1. Keep a clean upstream tracking branch

* Have one branch that mirrors upstream main and never contains your custom commits.
* Example branch name: upstream-main.

2. Do all your work in your own integration branch

* Create a long-lived branch from upstream-main, for example agenscience-main.
* Build features on short-lived branches off agenscience-main, then merge them back into agenscience-main.

3. Prevent accidental push to Shopify remote

* If your current remote points to Shopify, disable push on that remote.
* Command:
  git remote set-url --push origin DISABLED
* Result: fetch/pull still works, push fails safely.

4. Pull upstream changes safely

* Regularly fetch upstream.
* Update upstream-main fast-forward only.
* Then bring updates into agenscience-main with either:

1. merge (safer history, fewer surprises), or
2. rebase (cleaner linear history).

* For ongoing product work, merge is usually the least risky.

5. Keep feature work isolated

* Branch pattern:

1. feat/voice-input
2. feat/cart-tools
3. fix/stream-timeout

* Merge each branch into agenscience-main after testing.
* This keeps rollback easy and helps you track what changed.

6. Use local backups since you do not plan to push

* Periodically create a full backup bundle:
  git bundle create ../shop-chat-agent-backup-YYYYMMDD.bundle --all
* This gives you restore points without any hosted remote.

7. Optional but very useful: local mirror remote

* Create a local bare repo on disk and push to it as your private backup remote.
* You still avoid pushing to Shopify, but you get fast local recovery.

Minimal command pattern

* One-time safety:
  git remote set-url --push origin DISABLED
* Regular update cycle:
  git fetch origin
  git switch upstream-main
  git merge --ff-only origin/main
  git switch agenscience-main
  git merge upstream-main
* New feature:
  git switch -c feat/my-feature agenscience-main
  (work, commit)
  git switch agenscience-main
  git merge --no-ff feat/my-feature

Why this is best for your case

* You get upstream improvements continuously.
* Your custom enhancements stay clearly separated.
* You avoid accidental pushes to the reference repo.
* You keep a maintainable history even if everything stays local.
